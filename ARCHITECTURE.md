# Architecture Overview

## Layered Design

### 1. Database Layer (`db/`)

**Database Manager** (`db/manager.py`)
- psycopg2 connection pooling
- Dependency injection pattern
- Auto commit on success / auto rollback on error
- Context manager for connection/cursor lifecycle
- FastAPI-style `get_db_connection()` / `get_db_cursor()` dependency

**Models** (`db/models/`)
- Pydantic models for data validation and serialization
- NOT SQLAlchemy ORM - database operations use raw SQL
- Split by entity: `customer.py`, `transaction.py`, `alert.py`, `report.py`
- **Feature Grouping** - Customer models grouped by domain for query efficiency
- **Flexible Schema** - Feature groups use `extra='allow'` for dynamic ML features
  - Core models (CustomerBasic): Explicitly define all fields for type safety
  - Feature groups: Define only `cif_no`, allow extras from feature engineering pipeline
  - Source of truth: `feature_catalog.json`, not code
  - Why? ML features change frequently, models should be flexible
- Separate Create/Update models for API operations

**Migrations** (`migrations/`)
- Alembic for database migrations
- Raw SQL migrations (not autogenerated from ORM)
- Version control for schema changes

**Repositories** (`db/repositories/`)
- Data access layer
- One repository per entity
- Encapsulates all SQL queries
- Returns domain objects

**Services** (`db/services/`)
- Business logic layer
- Orchestrates multiple repositories
- Handles Redis caching
- Formats data for LLM consumption
- Generates compliance reports

### 2. Tools Layer (`tools/`)

Wraps service layer functions as LangChain tools that agents can call:
- `db_tools.py` - Query customer data, transactions, alerts
- `cache_tools.py` - Manage Redis cache operations

### 3. Agent Layer (`agents/`)

LangGraph-orchestrated specialized agents:

**Coordinator Agent** (`coordinator.py`)
- Central orchestrator
- Routes requests to specialized agents
- Aggregates responses

**Intent Mapping Agent** (`intent_mapper.py`)
- Understands natural language queries
- Maps to database columns using feature catalog
- Extracts parameters (CIF numbers, date ranges, thresholds)

**Data Retrieval Agent** (`data_retrieval.py`)
- Executes queries via tools
- Uses Redis caching
- Returns formatted data

**Compliance Expert Agent** (`compliance_expert.py`)
- Provides procedural guidance
- Generates compliance reports
- Maps to AML typologies

### 4. UI Layer (`ui/`)

**Streamlit App** (`streamlit_app.py`)
- Chat interface
- Session management
- Conversation history

## Feature Grouping Optimization

### Problem: Monolithic Models Are Inefficient

With 40+ engineered features per customer, fetching all columns for every query is wasteful:
- **Database**: Unnecessary I/O for unused columns
- **LLM Context**: Wasted tokens on irrelevant features
- **Agent Performance**: Slower reasoning with excessive data

### Solution: Domain-Based Feature Groups

Customer data is split into logical groups based on use case:

#### **CustomerBasic** (9 fields)
```python
# Use for: Quick lookups, risk checks, profile display
# Query cost: ~100 LLM tokens
cif_no, name, country, kyc_status, risk_score, account_opened_date, ...
```

#### **CustomerTransactionFeatures** (15 fields)
```python
# Use for: Volume analysis, transaction patterns, velocity
# Time windows: w0_30, w0_90, w30_60, w0_180
sum_txn_count_w0_30, sum_txn_amount_w0_90, avg_txn_amount_w0_30, ...
```

#### **CustomerRiskFeatures** (4 fields)
```python
# Use for: Typology detection (structuring, high-risk countries)
count_high_risk_countries_w0_90, count_cash_intensive_txn_w0_90, ...
```

#### **CustomerBehavioralFeatures** (3 fields)
```python
# Use for: Channel analysis, velocity, anomalies
count_atm_withdrawals_w0_30, velocity_score_w0_30, ...
```

#### **CustomerNetworkFeatures** (4 fields)
```python
# Use for: Network analysis, community detection
# Source: Graph database
network_degree_centrality, count_unique_counterparties_w0_90, ...
```

#### **CustomerKnowledgeGraphFeatures** (3 fields)
```python
# Use for: PEP screening, sanctions, adverse media
# Source: External knowledge graph
pep_exposure_score, adverse_media_score, sanction_list_proximity
```

#### **CustomerFull** (40+ fields)
```python
# Use sparingly: Only when truly need everything
# Includes all feature groups + model scores
```

### Benefits

**1. Query Performance**
```python
# Bad: Always fetch everything
SELECT * FROM customers WHERE cif_no = 'C123'  # 40+ columns

# Good: Fetch only what's needed
SELECT cif_no, name, risk_score FROM customers WHERE cif_no = 'C123'  # 3 columns
```

**2. LLM Token Efficiency**
```python
# User: "What's the risk score for C123?"
# Agent needs: CustomerBasic (9 fields, ~100 tokens)
# NOT: CustomerFull (40+ fields, ~800 tokens)

# User: "Show network analysis for C123"
# Agent needs: CustomerBasic + CustomerNetworkFeatures (~13 fields, ~250 tokens)
# NOT: All 40+ fields
```

**3. Intent Mapping Optimization**

The Intent Mapper can route to specific feature groups:
```python
"show me transaction volume" → CustomerTransactionFeatures
"check for PEP exposure" → CustomerKnowledgeGraphFeatures
"analyze network connections" → CustomerNetworkFeatures
"get customer profile" → CustomerBasic
```

**4. Realistic ML Pattern**

Matches how features are engineered in production:
- Time-windowed aggregations (w0_30, w0_90, w0_180)
- Domain-specific features (transaction, behavioral, network)
- Multi-source features (internal DB, graph, knowledge graph)

### Repository Implementation

```python
class CustomerRepository:
    def get_basic(self, conn, cif_no: str) -> Optional[CustomerBasic]:
        """Fast query for basic profile."""
        with conn.cursor(cursor_factory=RealDictCursor) as cur:
            cur.execute(
                "SELECT id, cif_no, name, country, kyc_status, risk_score, "
                "account_opened_date, created_at, updated_at "
                "FROM customers WHERE cif_no = %s",
                (cif_no,)
            )
            row = cur.fetchone()
            return CustomerBasic(**row) if row else None

    def get_with_transaction_features(self, conn, cif_no: str) -> dict:
        """Get basic + transaction features."""
        basic = self.get_basic(conn, cif_no)

        with conn.cursor(cursor_factory=RealDictCursor) as cur:
            cur.execute(
                "SELECT cif_no, sum_txn_count_w0_30, sum_txn_amount_w0_90, "
                "avg_txn_amount_w0_30, max_single_txn_w0_180, ... "
                "FROM customers WHERE cif_no = %s",
                (cif_no,)
            )
            txn_row = cur.fetchone()

        return {
            "basic": basic,
            "transaction_features": CustomerTransactionFeatures(**txn_row)
        }
```

## Flexible Feature Schema Pattern

### Problem: Maintaining ML Features in Code

ML systems have dozens or hundreds of engineered features that:
- Change frequently as feature engineering evolves
- Come from multiple sources (DB, graph, external APIs)
- Are domain-specific and time-windowed
- Would create massive maintenance overhead if hardcoded

**Bad Approach:**
```python
class CustomerTransactionFeatures(BaseModel):
    cif_no: str
    sum_txn_count_w0_30: int  # What if we add w0_7, w0_14, w0_21?
    sum_txn_amount_w0_30: Decimal  # What if we add median, mode?
    avg_txn_amount_w0_30: Decimal  # 50+ fields to maintain in code
    max_txn_amount_w0_30: Decimal
    # ... 40 more fields ...
    # Every feature engineering change = code change
```

### Solution: Feature Groups with Flexible Schema

**Core models** - Explicitly define stable fields:
```python
class CustomerBasic(BaseModel):
    """Stable identity fields that rarely change."""
    model_config = ConfigDict(from_attributes=True)

    id: Optional[int]
    cif_no: str
    name: str
    country: Optional[str]
    kyc_status: Optional[str]
    risk_score: Optional[float]
    # These don't change frequently, safe to hardcode
```

**Feature groups** - Minimal + flexible:
```python
class CustomerFeatureGroup(BaseModel):
    """Base class for ML feature groups."""
    model_config = ConfigDict(
        from_attributes=True,
        extra='allow',  # Accept any additional fields
    )
    cif_no: str  # Only required field

class CustomerTransactionFeatures(CustomerFeatureGroup):
    """Transaction features - dynamically loaded from DB.

    Example features (see feature_catalog.json for complete list):
    - sum_txn_count_w0_30, avg_txn_amount_w0_90
    - max_single_txn_w0_180, velocity_score_w0_30

    Note: Actual fields are dynamic, validated by catalog, not code.
    """
    pass  # No explicit field definitions!
```

### Benefits

**1. Flexibility**
```python
# Add new feature to database + catalog
ALTER TABLE customers ADD COLUMN sum_txn_count_w0_7 INTEGER;

# feature_catalog.json
{
  "sum_txn_count_w0_7": {
    "type": "integer",
    "description": "Weekly transaction count",
    ...
  }
}

# Code automatically supports it - no changes needed!
features = CustomerTransactionFeatures(**db_row)
print(features.sum_txn_count_w0_7)  # Works!
```

**2. Source of Truth = Feature Catalog**
```python
# feature_catalog.json defines ALL features
{
  "customers": {
    "feature_groups": {
      "transaction_features": {
        "columns": {
          "sum_txn_count_w0_30": {...},
          "avg_txn_amount_w0_90": {...},
          // Hundreds of features defined here, not in code
        }
      }
    }
  }
}
```

**3. Validation Still Works**
```python
# Required fields validated
CustomerTransactionFeatures(cif_no="C123", sum_txn_count_w0_30=50)  # ✅

# Missing required field
CustomerTransactionFeatures(sum_txn_count_w0_30=50)  # ❌ ValidationError

# Extra fields accepted
CustomerTransactionFeatures(
    cif_no="C123",
    some_new_ml_feature=0.75  # ✅ Allowed via extra='allow'
)
```

**4. Clean Separation**
- **Pydantic models**: Structure validation (not schema definition)
- **Feature catalog**: Complete feature definitions for LLM/Intent Mapper
- **Database schema**: Actual column definitions (Alembic migrations)


## Feature Group Caching Strategy

### The Problem with Naive Caching

**Bad Approach 1: Cache entire customer**
```python
cache_key = f"customer:{cif_no}"
# Problem: 40+ fields cached, but user only needs risk_score
# Wastes Redis memory, slower serialization
```

**Bad Approach 2: Cache individual columns**
```python
cache_key = f"customer:{cif_no}:sum_txn_count_w0_30"
# Problem: 15 separate cache lookups for transaction features
# Network overhead, cache key explosion
```

### Solution: Cache at Feature Group Granularity

**Feature groups = natural cache boundaries**

```python
# Cache keys by feature group
"customer:{cif_no}:basic"                  # CustomerBasic
"customer:{cif_no}:transaction_features"   # CustomerTransactionFeatures
"customer:{cif_no}:risk_features"          # CustomerRiskFeatures
"customer:{cif_no}:behavioral_features"    # CustomerBehavioralFeatures
"customer:{cif_no}:network_features"       # CustomerNetworkFeatures
"customer:{cif_no}:knowledge_graph"        # CustomerKnowledgeGraphFeatures
```

### Cache Hit Optimization Example

**Scenario:**
```python
# Query 1: User asks "What's the transaction count for C123?"
# - Intent mapper routes to: CustomerTransactionFeatures
# - Service layer checks cache: MISS
# - Fetch from DB: All 15 transaction features
# - Cache the entire group: "customer:C123:transaction_features"

# Query 2 (5 seconds later): "What's the avg transaction amount for C123?"
# - Intent mapper routes to: CustomerTransactionFeatures
# - Service layer checks cache: HIT! ✅
# - Return from Redis (no DB query)
# - Saved: 1 DB query, ~200ms latency

# Query 3 (10 seconds later): "Show me the max transaction for C123"
# - Intent mapper routes to: CustomerTransactionFeatures
# - Service layer checks cache: HIT! ✅
# - All transaction feature queries now cache hits
```

### Benefits

**1. Optimal Cache Hit Rate**
- User asks for ANY field in a group → entire group cached
- Subsequent queries for other fields in same group → cache hit
- Example: 15 transaction features = 1 cache entry, not 15

**2. Reduced Database Load**
```python
# Without grouping: 3 separate queries
SELECT sum_txn_count_w0_30 FROM customers WHERE cif_no = 'C123'
SELECT avg_txn_amount_w0_90 FROM customers WHERE cif_no = 'C123'
SELECT max_single_txn_w0_180 FROM customers WHERE cif_no = 'C123'

# With grouping + caching: 1 query, 2 cache hits
SELECT [all transaction features] FROM customers WHERE cif_no = 'C123'  # Query 1
# Queries 2 & 3: Redis cache hits
```

**3. Intelligent TTL Strategy**

Different feature groups have different update frequencies:

```python
CACHE_TTL = {
    "basic": 3600,                    # 1 hour (KYC, identity changes rarely)
    "transaction_features": 300,      # 5 minutes (updated frequently)
    "risk_features": 300,             # 5 minutes (recalculated often)
    "behavioral_features": 600,       # 10 minutes (medium frequency)
    "network_features": 1800,         # 30 minutes (graph analysis is expensive)
    "knowledge_graph": 7200,          # 2 hours (external data, slow to change)
}
```

**4. Memory Efficiency**

```python
# CustomerBasic: ~500 bytes
# CustomerTransactionFeatures: ~800 bytes
# Total for all groups: ~3KB per customer

# vs. Full customer always cached: ~5KB per customer
# Savings: 40% memory reduction when only partial data needed
```

### Service Layer Implementation

```python
class DataService:
    def __init__(self, customer_repo, cache_service):
        self.customer_repo = customer_repo
        self.cache = cache_service

    def get_customer_transaction_features(self, cif_no: str) -> CustomerTransactionFeatures:
        """Get transaction features with caching."""
        # Check cache first
        cache_key = f"customer:{cif_no}:transaction_features"
        cached = self.cache.get(cache_key)

        if cached:
            return CustomerTransactionFeatures(**cached)

        # Cache miss - fetch from database
        with db_manager.get_connection() as conn:
            features = self.customer_repo.get_transaction_features(conn, cif_no)

        # Cache the entire feature group
        self.cache.set(
            cache_key,
            features.model_dump(),
            ttl=300  # 5 minutes
        )

        return features

    def get_customer_profile(self, cif_no: str, include_groups: list[str]) -> dict:
        """Get customer with specific feature groups, using cache."""
        result = {}

        # Always get basic info
        result["basic"] = self._get_cached_or_fetch(
            cif_no, "basic", self.customer_repo.get_basic
        )

        # Fetch requested groups
        if "transaction_features" in include_groups:
            result["transaction_features"] = self._get_cached_or_fetch(
                cif_no, "transaction_features",
                self.customer_repo.get_transaction_features
            )

        if "network_features" in include_groups:
            result["network_features"] = self._get_cached_or_fetch(
                cif_no, "network_features",
                self.customer_repo.get_network_features
            )

        return result

    def _get_cached_or_fetch(self, cif_no: str, group_name: str, fetch_func):
        """Generic cache-or-fetch for any feature group."""
        cache_key = f"customer:{cif_no}:{group_name}"
        cached = self.cache.get(cache_key)

        if cached:
            return cached

        # Fetch from DB
        with db_manager.get_connection() as conn:
            data = fetch_func(conn, cif_no)

        # Cache it
        self.cache.set(cache_key, data.model_dump(), ttl=self._get_ttl(group_name))
        return data

    def _get_ttl(self, group_name: str) -> int:
        """Get TTL based on feature group update frequency."""
        return CACHE_TTL.get(group_name, 600)
```

### Cache Invalidation Strategy

**Event-driven invalidation:**

```python
# When customer data updated
def update_customer_risk_score(cif_no: str, new_score: float):
    with db_manager.get_connection() as conn:
        # Update database
        repo.update_risk_score(conn, cif_no, new_score)

    # Invalidate affected cache groups
    cache.delete(f"customer:{cif_no}:basic")  # Contains risk_score
    # Transaction features still cached - not affected

# When new transactions added
def add_transaction(transaction: TransactionCreate):
    with db_manager.get_connection() as conn:
        repo.create_transaction(conn, transaction)

    # Invalidate transaction-related caches
    cache.delete(f"customer:{transaction.customer_id}:transaction_features")
    cache.delete(f"customer:{transaction.customer_id}:risk_features")
    # Basic info still cached - not affected
```

## Data Schema

### Feature Engineering Patterns

AML models use heavily engineered features following naming conventions:

**Pattern:** `{aggregation}_{feature}_{timewindow}`

**Examples:**
- `sum_txn_count_w0_30` - Sum of transaction count, 0-30 days
- `avg_txn_amount_w0_90` - Average transaction amount, 0-90 days
- `max_single_txn_w0_180` - Max single transaction, 0-180 days
- `count_high_risk_countries_w0_90` - Count of high-risk country transactions

**Aggregation Types:**
- `sum_` - Summation
- `avg_` - Average
- `max_` - Maximum
- `min_` - Minimum
- `std_` - Standard deviation
- `count_` - Count
- `ratio_` - Ratio/percentage

**Time Windows:**
- `w0_30` - Last 0-30 days
- `w0_90` - Last 0-90 days
- `w30_60` - Days 30-60 ago
- `w0_180` - Last 0-180 days

**Network Features:**
- `network_degree_centrality` - Graph centrality score
- `network_community_id` - Community detection ID
- `count_unique_counterparties_w0_90` - Unique transaction partners
- `count_shared_counterparties` - Shared network connections

**Knowledge Graph Features:**
- `pep_exposure_score` - Politically Exposed Person proximity
- `adverse_media_score` - Negative news associations
- `sanction_list_proximity` - Distance to sanctioned entities

### Database Models

**Customer** - Main entity with `cif_no` (Customer Information File)
- Basic info: name, DOB, country, KYC status
- 40+ engineered risk features
- Model scores: ML, rule-based, overall risk

**Transaction** - Financial transactions
- Amount, currency, date, type, channel
- Counterparty details
- Risk flags: cash, round amount, high-risk country

**Alert** - AML alerts and suspicious activity
- Alert type (typology), severity, status
- Investigation tracking
- Model confidence and feature importance

**Report** - Compliance reports (SAR, STR, internal)
- Report content and narrative
- Approval workflow
- Filing status

## Repository Pattern with Raw SQL

```python
from psycopg2.extensions import connection as PGConnection
from psycopg2.extras import RealDictCursor
from typing import List, Optional
from db.models import CustomerModel

class CustomerRepository:
    """Data access layer for customers using raw SQL."""

    def get_by_cif(self, conn: PGConnection, cif_no: str) -> Optional[CustomerModel]:
        """Get customer by CIF number."""
        with conn.cursor(cursor_factory=RealDictCursor) as cur:
            cur.execute(
                "SELECT * FROM customers WHERE cif_no = %s",
                (cif_no,)
            )
            row = cur.fetchone()
            return CustomerModel(**row) if row else None

    def get_high_risk_customers(self, conn: PGConnection, threshold: float) -> List[CustomerModel]:
        """Get all customers above risk threshold."""
        with conn.cursor(cursor_factory=RealDictCursor) as cur:
            cur.execute(
                "SELECT * FROM customers WHERE risk_score > %s ORDER BY risk_score DESC",
                (threshold,)
            )
            rows = cur.fetchall()
            return [CustomerModel(**row) for row in rows]

    def create(self, conn: PGConnection, customer: CustomerModel) -> CustomerModel:
        """Create a new customer."""
        with conn.cursor(cursor_factory=RealDictCursor) as cur:
            cur.execute(
                """
                INSERT INTO customers (cif_no, name, country, kyc_status)
                VALUES (%s, %s, %s, %s)
                RETURNING *
                """,
                (customer.cif_no, customer.name, customer.country, customer.kyc_status)
            )
            row = cur.fetchone()
            return CustomerModel(**row)
```

# Multi-Agent System Deep Dive

## What is "State"?

**State** is a shared dictionary that gets passed between agents as they work on a user's query. Think of it like a **clipboard that agents pass around**.

### State Lifecycle Example

```python
# 1. User asks question
Initial State = {
    "user_query": "What's the risk score for C000001?",
    "messages": [],
    "next_agent": "coordinator",
    "intent": None,
    "retrieved_data": None,
    "final_response": None
}

# 2. Coordinator reads state and updates it
After Coordinator = {
    "user_query": "What's the risk score for C000001?",
    "next_agent": "intent_mapper",  ← Updated by Coordinator
    "messages": [{"role": "assistant", "content": "[Coordinator] Routing to intent_mapper"}]
}

# 3. Intent Mapper reads state and adds intent
After Intent Mapper = {
    "user_query": "What's the risk score for C000001?",
    "next_agent": "data_retrieval",
    "intent": {  ← Added by Intent Mapper
        "tools_to_use": ["get_customer_basic_info"],
        "entities": {"cif_no": "C000001"}
    }
}

# 4. Data Retrieval reads intent from state and adds data
After Data Retrieval = {
    "user_query": "What's the risk score for C000001?",
    "next_agent": "compliance_expert",
    "intent": {...},
    "retrieved_data": {  ← Added by Data Retrieval
        "success": true,
        "data": {"cif_no": "C000001", "risk_score": 27.15, ...}
    }
}

# 5. Compliance Expert reads everything and produces final answer
After Compliance Expert = {
    "user_query": "What's the risk score for C000001?",
    "intent": {...},
    "retrieved_data": {...},
    "compliance_analysis": {  ← Added by Compliance Expert
        "analysis": "Customer has low-medium risk...",
        "typologies": []
    },
    "final_response": "Customer C000001 has a risk score of 27.15...",  ← Final answer
    "completed": True
}
```

**Key Points:**
- Each agent **reads** from state
- Each agent **updates** state with its results
- State persists across the entire workflow
- State is the **single source of truth**

## Agent Architecture

### Agent-Tool Assignment

**Current Design:**
```
┌──────────────────────┐
│  Coordinator Agent   │  Tools: None (just routing logic)
└──────────────────────┘

┌──────────────────────┐
│ Intent Mapping Agent │  Tools: None (just LLM analysis)
└──────────────────────┘

┌──────────────────────┐
│ Data Retrieval Agent │  Tools: ALL 17 tools
└──────────────────────┘  - 8 customer tools
                          - 4 transaction tools
                          - 5 alert tools

┌──────────────────────┐
│Compliance Expert Agent│ Tools: None (just AML expertise)
└──────────────────────┘
```

**Tool Ownership:**
- **Only** the Data Retrieval Agent has access to tools
- Intent Mapper **decides** which tools to use (planning)
- Data Retrieval Agent **executes** those tools (execution)
- This separation keeps concerns clean: Planning vs Execution

**Where is this defined?**

In `agents/data_retrieval.py`:
```python
class DataRetrievalAgent:
    def __init__(self):
        # This line loads ALL 17 tools
        self.tools = get_all_tools()  ← All tools loaded here
        self.tool_map = {tool.name: tool for tool in self.tools}
```

### Detailed Agent Responsibilities

#### 1. Coordinator Agent (`agents/coordinator.py`)

**Role:** Router / Traffic Controller

**Input:** User query
**Output:** Routing decision

**Decision Logic:**
- "Show me customer data" → Route to Intent Mapper
- "What is structuring?" → Route directly to Compliance Expert (no data needed)
- "Analyze this customer" → Route to Intent Mapper (need data first)

**No tools** - Just uses LLM (GPT-4o-mini) to decide routing

**Example:**
```python
User: "What is the risk score for C000001?"
Coordinator thinks: "This needs customer data → route to intent_mapper"

State update:
{
    "next_agent": "intent_mapper"
}
```

#### 2. Intent Mapping Agent (`agents/intent_mapper.py`)

**Role:** Query Translator

**Input:** User query
**Output:** Structured intent (which tools, what params)

**No tools** - Uses LLM (GPT-4o-mini) + regex to:
1. Extract entities (CIF numbers, dates, amounts)
2. Map to specific feature groups
3. Identify which tools to call
4. Prepare tool arguments

**Example:**
```python
User: "Show me high-risk transactions for C000001"

Intent Mapper analyzes and outputs:
{
    "intent_type": "data_query",
    "entities": {"cif_no": "C000001"},
    "feature_groups": ["transaction_features"],
    "tools_to_use": [
        {
            "tool": "get_high_risk_transactions",
            "args": {"cif_no": "C000001", "limit": 10}
        }
    ],
    "confidence": 0.95
}
```

#### 3. Data Retrieval Agent (`agents/data_retrieval.py`)

**Role:** Tool Executor (No Interpretation!)

**Input:** Intent with tool specifications
**Output:** Raw factual data

**Has ALL 17 tools:**
```python
Customer Tools (8):
- get_customer_basic_info
- get_customer_transaction_features
- get_customer_risk_features
- get_customer_behavioral_features
- get_customer_network_features
- get_customer_knowledge_graph_features
- get_customer_full_profile
- search_customers_by_name

Transaction Tools (4):
- get_customer_transactions
- get_high_risk_transactions
- get_transaction_count
- get_transactions_by_date_range

Alert Tools (5):
- get_open_alerts
- get_alerts_by_severity
- get_alerts_by_type
- get_customer_alerts
- get_alert_details
```

**No LLM** - Just mechanically executes tools

**Critical:** Returns ONLY factual data:
- ✅ "Customer C000001 has risk_score: 27.15"
- ❌ "This customer has LOW risk" (that's interpretation - belongs in Compliance Expert)

**Example:**
```python
Input (from Intent Mapper):
{
    "tools_to_use": [
        {"tool": "get_customer_basic_info", "args": {"cif_no": "C000001"}}
    ]
}

Data Retrieval executes:
tool = self.tool_map["get_customer_basic_info"]
result = tool._run(cif_no="C000001")

Output:
{
    "success": True,
    "data": {
        "get_customer_basic_info": {
            "cif_no": "C000001",
            "name": "Christopher Davila",
            "risk_score": 27.15,
            "country": "CRI",
            ...
        }
    },
    "tools_used": ["get_customer_basic_info"]
}
```

#### 4. Compliance Expert Agent (`agents/compliance_expert.py`)

**Role:** AML Domain Expert / Analyst

**Input:** User query + Retrieved data (if any)
**Output:** Expert analysis, typologies, recommendations

**No tools** - Uses LLM (GPT-4o - more capable model) for:
- Interpreting transaction patterns
- Identifying AML typologies (structuring, layering, smurfing, etc.)
- Risk assessment
- Regulatory guidance (FATF, BSA/AML)
- Procedural recommendations (file SAR, escalate, EDD)

**Example:**
```python
Input:

Retrieved data: {

    "sum_txn_count_w0_90": 47,

Customer C000001 presents a low-medium AML risk profile...
```

## Why This Architecture?

### 1. Separation of Concerns
- **Planning** (Intent Mapper) vs **Execution** (Data Retrieval) vs **Analysis** (Compliance Expert)
- Each agent has ONE job
- Easy to test, debug, and improve each agent independently

### 2. Flexibility
- Intent Mapper can combine ANY tools as needed
- Can add new tools without changing agent code
- Different LLMs for different agents (cheap for routing, expensive for expertise)

### 3. Maintainability
- Tool assignment is explicit: Data Retrieval Agent owns ALL tools
- State is the single source of truth
- No hidden dependencies between agents

### 4. Performance
- Coordinator uses fast/cheap model (GPT-4o-mini)
- Intent Mapper uses fast/cheap model (GPT-4o-mini)
- Data Retrieval uses NO LLM (fast execution)
- Compliance Expert uses powerful model (GPT-4o) only when needed
